# 📋 采购需求池 - 编号说明文档

## 🔢 编号体系说明

### 1️⃣ **需求编号（Requirement Number）**
- **格式**：`QR-{区域}-{日期}-{随机数}`
- **示例**：`QR-SA-251217-1410`
- **含义**：这是**采购需求的内部编号**，由系统自动生成
- **用途**：
  - 采购员内部追踪采购需求
  - 关联采购订单(PO)
  - 采购部门内部流转使用

### 2️⃣ **来源单号（Source Reference）**
- **格式**：`RFQ-{区域}-{日期}-{随机数}` （业务员询价单号）
- **示例**：`RFQ-SA-251218-0001`
- **含义**：这是**业务员的询价单号**，表示这个采购需求从哪个业务员的询价单流转过来
- **用途**：
  - 追溯业务源头（但不显示客户信息）
  - 采购员知道这是哪个业务员发起的需求
  - 便于跨部门协作和沟通

## 🔒 权限隔离原则

### ✅ 采购员能看到：
- **需求编号** (QR-xxx) - 内部采购需求编号
- **来源单号** (RFQ-xxx) - 业务员询价单号（但不显示客户名称）
- **区域标签** (NA/SA/EA) - 知道服务哪个市场区域
- **产品信息** - 产品名称、型号、数量
- **供应商信息** - 供应商名称、联系方式、报价

### ❌ 采购员看不到：
- **客户公司名称** - 完全隐藏
- **客户联系人** - 完全隐藏
- **业务员姓名** - 完全隐藏
- **客户邮箱/电话** - 完全隐藏

### 🎯 设计目的：
1. **防止跳单**：采购员不知道客户是谁，无法直接联系客户
2. **保护商业机密**：客户资源完全由业务员掌控
3. **信息隔离**：通过RFQ编号关联业务流程，但不泄露客户身份
4. **市场区分**：通过区域标签(NA/SA/EA)帮助采购员了解市场需求

## ❓ 常见问题 FAQ

### Q1: 为什么需求编号和来源单号不一样？
**A:** 
- **需求编号 (QR-xxx)** 是采购部门的**内部编号**，用于追踪采购流程
- **来源单号 (RFQ-xxx)** 是**业务员的询价单号**，表示这个需求的业务源头
- 两者不同是为了区分**业务源头**和**采购流程**

### Q2: 为什么有时候两个编号一样？
**A:** 这是**旧测试数据**导致的。新系统已修复，正常情况下：
- 需求编号：`QR-SA-251217-1410` （采购内部编号）
- 来源单号：`RFQ-SA-251218-0001` （业务员询价单号）

### Q3: 为什么区域列是空的？
**A:** 这是**旧数据**没有region字段。新创建的采购需求都会自动带上区域信息（NA/SA/EA）。

### Q4: 我能看到客户名称吗？
**A:** **不能！** 这是严格的权限隔离原则。采购员只能看到：
- 来源单号（RFQ编号）
- 区域标签（NA/SA/EA）
- 产品信息
- **看不到任何客户信息**
