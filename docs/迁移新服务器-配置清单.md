# 迁移到新服务器（仅 IP 变化）— 配置清单

唯一变化是**新服务器 IP**（如从 `139.196.13.122` 换到 `47.82.231.239`），其他不变时，按下面清单检查。

---

## 一、前端（本仓库 innoshop_react）

| 位置 | 作用 | 当前值 |
|------|------|--------|
| **`.env.local`** | 本地开发时请求的 API 地址 | `VITE_API_BASE_URL=http://47.82.231.239` |
| **`.env.production`** | 打包部署后请求的 API 地址 | `VITE_API_BASE_URL=http://47.82.231.239` |
| **`vite.config.ts`** → `server.proxy['/api'].target` | 本地开发时代理目标（若用代理） | `http://47.82.231.239` |
| **`src/api/backend-auth.ts`** → `getApiBaseUrl()` 默认返回值 | 未配置环境变量时的兜底 API 地址 | `http://47.82.231.239` |

**说明：** 换新 IP 时，把上面 4 处里的旧 IP 全部改成新 IP 即可。前端没有别的“域名/指定 IP”配置。

---

## 二、新服务器上的配置（必做）

### 0. 先修 500：补上 Laravel APP_KEY（镜像后最常见）

**现象：** 直接访问新 IP 报 **500 Internal Server Error**，错误信息是  
`Illuminate\Encryption\MissingAppKeyException: No application encryption key has been specified.`

**原因：** 镜像复制时 `.env` 经常没被带上，或 `APP_KEY` 为空。Laravel 启动必须要有加密密钥。

**在新服务器上执行：**

```bash
cd /www/wwwroot/你的站点目录/api   # 实际 Laravel 项目根目录，例如 /www/wwwroot/47.82.231.239/api 或 /www/wwwroot/139.196.13.122/api
php artisan key:generate
```

会把 `APP_KEY=base64:xxx...` 自动写入 `.env`。若没有 `.env`，先复制再生成：

```bash
cp .env.example .env
php artisan key:generate
```

然后清配置缓存：

```bash
php artisan config:clear
php artisan cache:clear
```

再在浏览器访问 `http://47.82.231.239`，500 应消失；接口不再卡死。

---

### 1. 面板 / Nginx 站点

- **网站名 / server_name**：改成新 IP（如 `47.82.231.239`），或同时写旧 IP 和新 IP。
- **根目录**：指向 Laravel 的 `public`，例如  
  `/www/wwwroot/47.82.231.239/api/public`  
  或镜像复制后实际路径（如仍是 `/www/wwwroot/139.196.13.122/api/public`）。
- 修改后 **Nginx 重载**。

### 2. Laravel 镜像复制后必须重新配置的项（与 Nginx 无关）

在**新服务器**上，进入 Laravel 项目根目录（如 `/www/wwwroot/139.196.13.122/api` 或你的实际路径），按顺序做：

#### ① 环境与密钥

| 配置项 | 说明 | 操作 |
|--------|------|------|
| **`.env` 是否存在** | 镜像可能没带上 | 没有则：`cp .env.example .env` |
| **`APP_KEY`** | 必须存在且非空 | `php artisan key:generate` |
| **`APP_URL`** | 新站点地址（域名或 IP） | 改成 `https://api.cosunchina.com` 或当前访问地址 |
| **`APP_ENV`** | 生产环境 | 建议 `APP_ENV=production` |
| **`APP_DEBUG`** | 生产建议关闭 | `APP_DEBUG=false` |
| **`DB_HOST` / `DB_DATABASE` / `DB_USERNAME` / `DB_PASSWORD`** | 新机数据库 | 数据库在本机用 `127.0.0.1`，否则填实际主机 |

#### ② 清掉所有缓存（否则会沿用旧配置/路由）

```bash
cd /www/wwwroot/你的目录/api
php artisan config:clear
php artisan cache:clear
php artisan route:clear
php artisan view:clear
```

#### ③ 目录权限（否则可能 500 或写不进去）

```bash
chmod -R 775 storage bootstrap/cache
chown -R www:www storage bootstrap/cache   # 宝塔常见用户是 www，按你机器实际用户改
```

#### ④ 若请求一直“无响应”（例如 OPTIONS 卡住）

可能是**首请求**时 Laravel 用到了数据库（如 Session/Cache 驱动为 database），而新机数据库未连上或表不存在，导致请求挂起。可先改为文件驱动，确认能响应后再改回：

```env
SESSION_DRIVER=file
CACHE_STORE=file
```

改完 `.env` 后再执行一次：

```bash
php artisan config:clear
php artisan cache:clear
```

确认接口能正常返回后，再根据需要改回 `SESSION_DRIVER=database`、`CACHE_STORE=database`（并保证数据库和表已就绪）。

#### ⑤ 数据库与表（若使用 database 的 session/cache）

```bash
php artisan migrate --force
# 若 sessions 表缺失且 SESSION_DRIVER=database，会报错，可先改用 SESSION_DRIVER=file（见④）
```

### 3. 防火墙 / 安全组

- 新服务器**开放 80（和 443）**，允许你本机或用户访问。
- 若用云服务器，控制台里**安全组**也要放行 80/443。

---

## 三、本地请求“卡住、没反应”时排查

前端已经请求**指定 IP**（如 `http://47.82.231.239`）时，若一直无响应：

1. **本机能否访问新 IP**  
   本机执行：  
   `curl -v http://47.82.231.239/api/auth/login`  
   看是否超时、被拒绝或返回 4xx/5xx。

2. **新服务器上 Laravel 是否正常**  
   SSH 到新服务器：  
   `curl -v http://127.0.0.1/api/auth/login`  
   若本机 curl 超时但服务器本机 curl 正常，多半是**防火墙/安全组**未放行或只允许了旧 IP。

3. **数据库是否在新机、是否可达**  
   确认新机 `.env` 里 `DB_HOST`、`DB_DATABASE`、`DB_USERNAME`、`DB_PASSWORD` 正确；数据库若在别的机器，`DB_HOST` 填那台机器的 IP，并保证新服务器能访问该 IP 的 3306。

---

**总结：**  
“域名就是指定 IP”时，需要配置的只有：  
① 前端 4 处 API 地址（已改为新 IP）；  
② 新服务器 Nginx 站点名与根目录；  
③ 新服务器上 Laravel `.env` 的 `APP_URL`、`DB_HOST`；  
④ 新服务器防火墙/安全组放行 80/443。  
请求仍卡住时，按上面第三节逐项排查。
