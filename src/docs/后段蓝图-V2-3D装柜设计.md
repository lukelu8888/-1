# 后段蓝图 V2：3D装柜设计

## 1. 定位

本模块属于后段蓝图 V2 主线A当前段：

`完货确认`
→ `订舱 / Shipping Order`
→ `集货装柜计划`
→ `国内转运`
→ `仓库接货清点`
→ `待装柜池`
→ `3D装柜计算与装柜执行`

它不是单纯“画一个 3D 动画”，而是要解决真实业务问题：

- 根据外箱尺寸、重量、件数自动推荐柜型
- 自动测算最少柜数
- 自动把货分配到一个或多个柜
- 对每个柜做三维摆放方案
- 校验体积利用率、重量限制、重心、装柜顺序
- 将结果回写到 `container_load_plans / container_load_plan_items / loading_tasks`

## 2. 业务目标

系统需要同时满足 4 个层面：

1. 业务员 / 物流员
- 快速知道要用什么柜型、多少个柜

2. 仓库
- 提前知道哪票货进哪个柜、放在哪个区域

3. 装柜现场
- 按 3D 方案装柜，减少找货、错装、重装

4. 管理层
- 能看到柜利用率、超重风险、是否需要加柜

## 4. 输出结果

系统输出不能只是“推荐 40HQ 一柜”，而要输出 3 层结果：

### 4.1 柜型推荐层

- 推荐柜型：
  - `20GP`
  - `40GP`
  - `40HQ`
  - 后续可扩 `45HQ`
- 推荐柜数
- 每柜体积利用率
- 每柜重量利用率
- 是否存在超重风险

### 4.2 分柜方案层

对每条货输出：

- 分配到哪个柜
- 柜内顺序
- 预计放置区
- 是否需要调整到下一柜

### 4.3 3D摆放层

对每个柜输出：

- 每个箱体的 `x / y / z`
- 朝向 `rotation`
- 所属货物
- 所属供应商
- 颜色分组
- 装载顺序

- 内长约 `1203cm`

## 6. 计算逻辑

建议分 3 层算法，不要一步到位做超复杂最优解。

### 6.1 第一层：快速估柜

目的：

- 快速给业务员一个初步答案

算法：

1. 汇总总 CBM
2. 汇总总毛重
3. 分别按柜体积和载重做上取整
4. 取更大的那个结果
5. 对 `20GP / 40GP / 40HQ` 分别算一次
6. 给出推荐值

输出：

- 推荐柜型
- 推荐柜数
- 体积利用率
- 重量利用率

这个阶段还不做真实 3D，只做“快速预估”。

### 6.2 第二层：规则分柜

目的：

- 把货真正分配到多个柜

规则优先级建议：

1. 不得超载重
2. 不得超可用体积
3. 同供应商货物尽量集中
4. 同一集货计划项尽量不拆柜
5. 易碎、竖放货优先单独处理
6. 重货优先靠前靠下
7. 后装货优先靠门

输出：

- 每个柜有哪些货
- 每个柜预计剩余空间

### 6.3 第三层：3D装箱

目的：

- 真正生成三维摆放方案

建议采用：

- `3D Bin Packing`
- 先用启发式算法，不先做最优求解器

推荐算法思路：

1. 箱体按“底面积大、重量大、体积大”排序
2. 每次优先放到当前柜的最低可用位
3. 允许有限旋转：
  - `LWH`
  - `WLH`
  - `HLW`
  - 但受 `upright_only` 约束
4. 放置时检查：
  - 是否越界
  - 是否碰撞
  - 下方支撑是否足够
  - 层高限制是否满足
  - 局部承重是否超限
5. 放不下则换下一个空位或下一柜

## 7. 3D呈现建议

当前仓库没有 `three.js` 或 `@react-three/fiber`，所以：

- 现在可以先完成“算法 + 2D平面/剖面图”
- 真正 3D 呈现建议下一阶段引入：
  - `three`
  - `@react-three/fiber`
  - `@react-three/drei`

### 7.1 视图建议

最少做 4 个视图：

1. 柜体 3D 视图
- 旋转
- 缩放
- 平移

2. 正视图
- 看堆高

3. 俯视图
- 看平面占用

4. 装柜顺序视图
- 按顺序高亮第 1、2、3 步

### 7.2 颜色规则

- 同供应商同色系
- 不同产品不同深浅
- 异常箱体高亮红色
- 易碎品橙色
- 竖放限制品蓝色

### 7.3 交互

- 点击箱体显示：
  - 产品
  - 供应商
  - 件数
  - 重量
  - 坐标
- 支持隐藏某供应商
- 支持只看某个柜
- 支持导出静态装柜图

## 9. 与现有主线挂接

### 9.1 触发时机

建议触发点：

- 在 `Cargo Receipt` 完成后
- 且集货计划状态达到 `ready_for_loading`

也就是：

`集货计划 ready_for_loading`
→ `生成装柜计算方案`
→ `人工确认`
→ `创建 container_load_plan`
→ `创建 loading_tasks`

### 9.2 回写对象

算法结果最终回写：

- `container_load_plans`
- `container_load_plan_items`
- `loading_tasks`

### 9.3 与仓库的关系

仓库接货时就要知道：

- 哪票货进哪个柜
- 仓内分区
- 装柜顺序

所以 3D 装柜方案不是装柜当天才算，最好在：

- 集货基本完成后
- 正式装柜前

先生成一版预案。

## 10. MVP建议

第一期不要一上来就做“最优 3D AI 装柜”。

### MVP-1

- 快速估柜
- 推荐柜型/柜数
- 分柜
- 2D平面图

### MVP-2

- 3D摆放
- 碰撞检测
- 旋转视图
- 导出装柜图

### MVP-3

- 多柜联合优化
- 多点装柜顺序优化
- 重心平衡
- AI调参 / 多方案对比

## 11. 当前建议的实施顺序

按轻重缓急，我建议：

1. 先做 `container_type_specs`
2. 再做 `快速估柜算法`
3. 再做 `分柜算法`
4. 再做 `2D待装柜/分柜图`
5. 最后做 `3D可视化`

原因：

- 先把“能不能装下、要几个柜”算对，比先把 3D 画漂亮更重要
- 先让仓库知道如何分货，比先做旋转缩放更有业务价值

## 12. 结论

这个模块在 ERP 里的正确定位应是：

**装柜决策与3D执行模块**

不是单纯可视化工具，也不是纯物流附件。

它同时承担：

- 估柜
- 分柜
- 分区堆货
- 装柜执行指导
- 装柜复盘留痕

后续正式实现时，建议作为后段蓝图 V2 当前段的核心模块推进。
