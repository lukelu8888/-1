# 业务员待办中心 V1 服务层草案

## 一、目的

这份服务层草案只服务于：

- **业务员待办中心 V1**

目标是明确：

1. 服务层从哪些现有业务对象取数
2. 如何把真实业务节点聚合成业务员待办
3. 如何把业务员动作回写到行为层和业务层
4. 页面最终拿到什么结构

本草案不涉及：

- 其它角色待办
- 通用任务中心
- 工作台数据分析

---

## 二、服务层定位

业务员待办中心的服务层不是一个“通用任务服务”。

它的定位是：

- 业务员客户跟进聚合服务
- 业务员动作写入服务
- 客户回复写入服务

建议按 3 层拆分：

1. `salesTodoQueryService`
2. `salesFollowUpService`
3. `salesCustomerReplyService`

---

## 三、服务层结构

## 3.1 salesTodoQueryService

### 作用

负责从真实业务链实时聚合出业务员待办页面所需数据。

### 输入

- 当前业务员身份
  - `sales_user_id`
  - `sales_user_email`
- 过滤条件
  - `bucket`
  - `keyword`
  - `date_range`
  - `priority_only`

### 输出

- 顶部动作条摘要
- 主执行表列表
- 右侧辅助栏数据
- 详情抽屉数据

### 不负责

- 直接写业务员跟进动作
- 直接写客户回复

---

## 3.2 salesFollowUpService

### 作用

负责写入业务员动作日志，并在必要时推动真实业务节点。

### 负责动作

- 已联系客户
- 已发 `QT`
- 已催合同
- 已催定金
- 已发验货报告
- 已催付款
- 已通知到港
- 已催清关资料
- 已确认收货
- 已邀请反馈
- 记录客户回复
- 约下次跟进

### 输出

- 新增一条 `sales_follow_up_logs`
- 必要时回写真实业务对象

---

## 3.3 salesCustomerReplyService

### 作用

负责记录客户真实回复，供“最近客户回复”和待办规则使用。

### 负责场景

- 客户确认 `QT`
- 客户要求改价
- 客户确认 `SC`
- 客户选择验货方式
- 客户安排第三方验货
- 客户安排第三方监装
- 客户承诺付款
- 客户确认到港
- 客户提供清关资料
- 客户确认收货
- 客户提交反馈

### 输出

- 新增一条 `sales_customer_reply_logs`
- 必要时推动对应业务状态

---

## 四、聚合输入对象

业务员待办中心 V1 的查询层建议从以下对象实时取数：

| 来源对象 | 用途 |
|---|---|
| `Inquiry` | `ING` 节点 |
| `QR` | 成本回传节点 |
| `SalesQuotation` | `QT` 节点 |
| `SalesContract` | `SC` 节点 |
| `purchase_order_execution` | 完货后客户协同节点 |
| `voyage_tracking` | 在途节点 |
| `arrival_notices` | 到港节点 |
| `import_clearance_coordination` | 清关资料协同 |
| `delivery_confirmations` | 收货确认节点 |
| `post_order_feedback` | 客户反馈节点 |
| `sales_follow_up_logs` | 最近联系 / 下次跟进 / 我的最近跟进 |
| `sales_customer_reply_logs` | 最近客户回复 |

## 六、节点聚合规则

## 6.1 ING 聚合规则

### 触发条件

- `Inquiry.isSubmitted = true`
- 指派给当前业务员
- `status` 仍处于待处理或待澄清

### 转换结果

生成：

- 新询价待响应
- 询价待澄清

### 主表建议动作

- 已联系客户
- 已完成澄清
- 已转 `QR`
- 约下次跟进

---

## 6.2 QR 聚合规则

### 触发条件

- 已存在 `QR`
- 成本回传未完成

### 转换结果

生成：

- 待内部成本反馈

### 主表建议动作

- 已催采购
- 已收成本反馈
- 约下次跟进

---

## 6.3 QT 聚合规则

### 触发条件

- 已批准未发送
- 已发送未反馈
- 协商中

### 转换结果

生成：

- 待发报价
- 待客户确认报价
- 待跟进客户改价意见

### 主表建议动作

- 已发 `QT`
- 已联系客户
- 客户要求改价
- 已催报价反馈
- 约下次跟进

---

## 6.4 SC 聚合规则

### 触发条件

- 已批准未发送
- 已发未签
- 已确认未付定金

### 转换结果

生成：

- 待发合同
- 待签合同
- 待客户定金

### 主表建议动作

- 已发 `SC`
- 已催签署
- 已催定金
- 约下次跟进

---

## 6.5 履约后客户协同规则

### 来源

- `purchase_order_execution`

### 触发条件

- 未确定验货方式
- 第三方验货未安排
- 第三方监装未安排
- 客户付款未完成
- 客户未确认海运费/船期

### 转换结果

- 待确认验货方式
- 待安排第三方验货
- 待安排第三方监装
- 待客户付款
- 待确认海运费/船期

### 主表建议动作

- 已联系客户
- 已发验货报告
- 已催付款
- 已确认海运费/船期
- 约下次跟进

---

## 6.6 交付后协同规则

### 来源

- `arrival_notices`
- `import_clearance_coordination`
- `delivery_confirmations`
- `post_order_feedback`

### 触发条件

- 到港通知已发未确认
- 清关资料未配合
- 收货未确认
- 反馈未提交

### 转换结果

- 待确认到港
- 待清关资料
- 待收货回执
- 待客户反馈

### 主表建议动作

- 已通知到港
- 已催清关资料
- 已确认收货
- 已邀请反馈
- 约下次跟进

---

## 七、优先级计算规则

建议由服务层统一输出优先级：

### 1. critical

- 已超期未联系
- 付款卡住放单
- 到港后客户长时间无回应

### 2. high

- 新 `ING`
- `QT` 已发未反馈超过阈值
- `SC` 已发未签超过阈值
- 定金/尾款承诺未兑现

### 3. medium

- 一般性跟进
- 已进入协商状态

### 4. normal

- 例行复联
- 反馈邀请

## 十、V1 建议实现顺序

### 第一步

实现只读聚合：

- `ING`
- `QT`
- `SC`
- 付款
- 到港
- 收货
- 反馈

### 第二步

实现动作写入：

- 已联系客户
- 已发 `QT`
- 已催合同
- 已催定金
- 已催付款
- 已通知到港
- 已确认收货
- 已邀请反馈
- 约下次跟进

### 第三步

实现客户回复写入：

- 报价反馈
- 合同确认
- 付款承诺
- 到港确认
- 收货确认
- 反馈提交

### 第四步

再补：

- 第三方验货
- 第三方监装
- 清关资料协同

## 十一、最终判断标准

如果服务层最终变成：

- 一套所有角色共用的通用任务中心服务

那就说明已经走偏。

如果服务层最终做到：

1. 只聚合业务员主链节点
2. 只输出业务员需要的执行信息
3. 动作日志和客户回复独立落库
4. 关键动作能回写真实业务表

那才说明服务层方向是对的。
