# 权限中心 V1 服务层草案

## 一、目标

为新建的 **权限中心 V1** 定义清晰的服务层拆分与接口结构，使后续实现时：

- 不继续往 `supabaseService.ts` 堆逻辑
- 各页签有清晰的数据读写职责
- 角色、菜单、动作、范围、例外、日志可独立维护

---

## 二、服务层拆分建议

建议拆成以下独立 service 文件：

- `permissionRoleService.ts`
- `permissionModuleService.ts`
- `permissionAccessService.ts`
- `permissionActionService.ts`
- `permissionScopeService.ts`
- `permissionUserExceptionService.ts`
- `permissionChangeLogService.ts`
- `permissionCenterQueryService.ts`

其中：
- 前 7 个负责单一表/单一层
- `permissionCenterQueryService` 负责页面聚合输出

---

## 三、各 service 职责

## 1. `permissionRoleService`

### 负责对象
- `permission_roles`

### 职责
- 读取正式角色清单
- 维护角色状态与排序
- 提供角色详情

### 建议方法
- `listRoles(filters?)`
- `getRoleByCode(roleCode)`
- `getRoleById(id)`
- `updateRoleStatus(id, status)`
- `updateRoleSortOrder(items)`
- `seedOfficialRoles()`

### 说明
- V1 不建议开放任意新增角色 UI
- 角色先以固定清单维护为主

---

## 2. `permissionModuleService`

### 负责对象
- `permission_modules`

### 职责
- 读取模块字典
- 维护模块层级和排序

### 建议方法
- `listModules(filters?)`
- `listModulesGroupedByCategory()`
- `getModuleByCode(moduleCode)`
- `updateModuleSortOrder(items)`
- `seedDashboardModules()`

### 说明
- 初始模块来源应与现有 AdminDashboard 对齐

---

## 3. `permissionAccessService`

### 负责对象
- `permission_role_module_access`

### 职责
- 管理角色到模块的菜单可见性

### 建议方法
- `listRoleModuleAccess(roleId?)`
- `getMatrix()`
- `setModuleVisible(roleId, moduleId, isVisible)`
- `batchSaveMatrix(payload)`
- `resetToDefaultMatrix()`

### 说明
- 这一层对应新权限中心的“菜单权限矩阵”
- 主要吸收旧图2

---

## 4. `permissionActionService`

### 负责对象
- `permission_role_module_actions`

### 职责
- 管理角色在模块内的动作权限

### 建议方法
- `listActionsByModule(moduleId)`
- `listActionsByRole(roleId)`
- `getRoleModuleActions(roleId, moduleId)`
- `setRoleModuleAction(roleId, moduleId, actionCode, isAllowed)`
- `batchSaveRoleModuleActions(payload)`

### 说明
- 这一层吸收旧图1中的动作图标能力

---

## 5. `permissionScopeService`

### 负责对象
- `permission_role_module_scopes`

### 职责
- 管理角色在模块内的数据范围

### 建议方法
- `listScopesByModule(moduleId)`
- `getRoleModuleScope(roleId, moduleId)`
- `setRoleModuleScope(roleId, moduleId, scopeCode, scopeRules?)`
- `batchSaveRoleModuleScopes(payload)`

### 说明
- 这一层是权限中心的“深度控制层”
- 必须和动作层分开

---

## 6. `permissionUserExceptionService`

### 负责对象
- `permission_user_exceptions`

### 职责
- 处理按人例外授权

### 建议方法
- `listExceptions(filters?)`
- `getExceptionById(id)`
- `createException(payload)`
- `updateException(id, payload)`
- `disableException(id)`
- `expireInvalidExceptions()`

### 说明
- 这一层承接旧图1中“按人差异”
- 但不让真实人员进入主矩阵

---

## 7. `permissionChangeLogService`

### 负责对象
- `permission_change_logs`

### 职责
- 记录并查询权限变更日志

### 建议方法
- `listLogs(filters?)`
- `createLog(payload)`
- `getLogsByTarget(targetType, targetKey)`

### 说明
- 所有写操作最终都应写日志

---

## 8. `permissionCenterQueryService`

### 负责对象
- 聚合前面 7 层 service 的数据

### 职责
- 为新权限中心页面提供结构化输出
- 避免页面内自行拼装大量数据

### 建议方法

#### 角色总览页
- `getRoleOverviewPageData()`

返回建议：
- 角色列表
- 类型统计
- 允许登录统计
- 需区域字段统计

#### 菜单矩阵页
- `getMenuMatrixPageData()`

返回建议：
- 角色列
- 模块分组
- 角色-模块矩阵值
- 未保存变更摘要（如有草稿机制）

#### 动作与数据范围页
- `getActionAndScopePageData(moduleCode?)`

返回建议：
- 模块树
- 当前模块的角色动作矩阵
- 当前模块的角色范围配置

#### 人员例外授权页
- `getUserExceptionPageData(filters?)`

返回建议：
- 人员例外列表
- 人员基础信息摘要
- 例外状态统计

#### 日志页
- `getChangeLogPageData(filters?)`

返回建议：
- 日志列表
- 操作人列表
- 变更类型统计

---

## 四、建议的写入链路

### 1. 菜单矩阵保存

页面
→ `permissionAccessService.batchSaveMatrix()`
→ `permissionChangeLogService.createLog()`

### 2. 动作权限保存

页面
→ `permissionActionService.batchSaveRoleModuleActions()`
→ `permissionChangeLogService.createLog()`

### 3. 数据范围保存

页面
→ `permissionScopeService.batchSaveRoleModuleScopes()`
→ `permissionChangeLogService.createLog()`

### 4. 人员例外授权保存

页面
→ `permissionUserExceptionService.createException()/updateException()`
→ `permissionChangeLogService.createLog()`

---

## 五、与企业主数据中心的联动建议

权限中心不直接管理：
- 员工主档
- 登录账号

但人员例外授权页需要调用“企业主数据中心的人/账号摘要”。

建议复用或新增只读查询服务：
- `staffDirectoryService`
- 或未来单独的：
  - `internalAccountDirectoryService`

用于在例外授权页里选择：
- 人员
- 所属部门
- 所属区域
- 当前角色

---

## 六、服务层返回结构建议

统一建议返回：

- 页面直接可用结构
- 不让 React 页面自己大量做字段重排

例如：

```ts
type PermissionMenuMatrixPageData = {
  roles: Array<{ id: string; code: string; nameZh: string }>
  moduleGroups: Array<{
    category: string
    modules: Array<{
      id: string
      code: string
      nameZh: string
      visibleByRole: Record<string, boolean>
    }>
  }>
}
```

说明：
- `permissionCenterQueryService` 负责组装
- 页面只负责展示与触发保存

## 七、V1 实现顺序建议

### 第一步
- `permissionRoleService`
- `permissionModuleService`
- `permissionAccessService`
- `permissionCenterQueryService.getRoleOverviewPageData()`
- `permissionCenterQueryService.getMenuMatrixPageData()`

### 第二步
- `permissionActionService`
- `permissionScopeService`
- `permissionCenterQueryService.getActionAndScopePageData()`

### 第三步
- `permissionUserExceptionService`
- `permissionChangeLogService`
- 对应查询聚合接口

---

## 八、V1 暂不做

不建议在第一版就做：
- 复杂审批工作流联动
- 实时 diff 引擎
- 自定义角色工厂
- 跨侧（客户/供应商/第三方）权限统一

先把我方侧权限中心 V1 做稳。

---

## 九、结论

权限中心 V1 服务层应采用：

`单层 service + 聚合 query service`

的结构。

也就是：
- 每一层权限数据单独管理
- 页面统一由聚合服务喂数据
- 所有变更都走日志

这样既符合大厂常见的权限中心后端分层逻辑，也能避免继续把逻辑堆到一个超大 service 文件里。
