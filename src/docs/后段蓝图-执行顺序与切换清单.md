## 3. 执行顺序

### 3.1 推荐顺序

严格按以下顺序执行：

1. `063`
2. `064`
3. `065`

原因：

- `063` 是主线A最前面的真实执行底座
- `064` 依赖 `063` 中的 `container_load_plans`
- `065` 依赖 `063` 中的装柜/发运对象

### 3.2 为什么不要跳着执行

如果先上 `064` 或 `065`：

- 会出现 packet 已有、但前段执行对象还不存在
- 或在途/到港数据已有、但装柜与 shipment 源数据未建
- 容易造成主线倒挂

所以：

- 后段蓝图必须先把 `CG后执行层` 建好
- 再接财务包
- 最后接开船后跟踪与交付

## 5. 真实风险点

### 风险1：主线A和主线B先后关系被打乱

表现：

- 先接 packet，再接执行主线

结果：

- 财务文件包会变成空壳
- D 槽位无法被真实业务事件填充

处理：

- 严格先接 `063`
- 再接 `064`

### 风险2：旧页面继续写 mock 数据

表现：

- 新表已落库，但页面仍只维护本地 mock state

结果：

- 新旧两套数据并存
- 业务人员误以为已上线，实际没有入库

处理：

- 页面切换必须按“写入优先”推进
- 先把创建/更新动作接到新表
- 再切换列表读取

### 风险3：把 `purchase_orders.status` 继续当执行状态使用

表现：

- 继续把生产、自检、QC、装柜等状态塞到旧 `purchase_orders.status`

结果：

- 合同状态与执行状态混淆
- 以后统计与权限都会变乱

处理：

- 合同级状态留在 `purchase_orders`
- 执行级状态统一放 `purchase_order_execution`

### 风险4：文档中心和财务包职责混淆

表现：

- 继续把 `DocumentCenter` 当成完整财务归档系统

结果：

- 模板输出和财务齐套管理耦合
- 多来源文件和版本确认无处落地

处理：

- `DocumentCenter` 继续做模板引擎
- `Finance Compliance Packet` 做案卷与核验

### 风险5：旧 workbench 被误当成真入口

表现：

- 单证员工作台、矩阵页、sandbox 继续承接真实操作

结果：

- 数据继续散落在 mock 层

处理：

- workbench 只保留参考与培训用途
- 真入口只落在真实业务模块

## 6. 页面切换顺序

### 6.1 第一阶段：只切写入，不切展示

优先改这些入口的“提交/保存”逻辑：

1. `PurchaseOrderManagementEnhanced`
2. 验货相关页面
3. 装柜与发运入口页

目标：

- 用户操作先真正写进新表
- 列表页可以暂时不展示全部新字段

### 6.2 第二阶段：切列表与详情读取

按顺序切：

1. CG执行详情
2. 国内转运列表
3. 装柜计划与装柜任务列表
4. 财务 `Compliance Packets`
5. 在途/到港/清关/交付视图

### 6.3 第三阶段：冻结旧 mock 数据源

对下列对象进行处理：

- 本地 mock shipment
- workbench mock docs
- demo timeline

处理方式：

- 不删页面
- 但停止作为真实入口
- 在代码注释或页面标识中明确 `demo-only`

## 7. 模块改造顺序

### 7.1 先改主线A

优先级：

1. `PurchaseOrderManagementEnhanced`
2. `InspectionManagementComplete` 或现有 QC 页
3. `ShippingDocumentManagement`

原因：

- 主线A是后段蓝图的业务推进层
- A 不通，B 只是空包

### 7.2 再改主线B

优先级：

1. `FinanceManagement`
2. `PaymentRecordManagement`
3. `FinanceContext`
4. `documentation-service`

原因：

- 这些模块需要消费 A 主线产生的数据和文件
