# 业务员待办中心 V1 数据模型草案

## 一、目标

这份数据模型草案只服务于：

- **创建业务员待办中心**

不服务于：

- 通用待办中心
- 财务待办中心
- 单证待办中心
- 采购待办中心

本草案的目标是把业务员待办中心需要的最小数据结构钉死，保证后续：

1. 能生成业务员待办
2. 能记录业务员跟进动作
3. 能回写真实业务节点
4. 能支撑页面主表、辅助栏、详情抽屉

---

## 二、设计原则

### 1. 先服务业务员，不做通用模型

业务员待办中心只围绕业务员主链：

`ING -> QR -> QT -> SC -> 客户付款/验货/监装 -> 到港/清关/收货 -> 反馈`

### 2. 以真实业务对象为主，不手工造孤立任务

业务员待办不是独立业务对象，而是从真实业务节点派生出来。

### 3. 跟进日志必须落库

业务员的动作不能只存在前端本地状态，必须有正式日志表。

### 4. 待办列表优先实时聚合

V1 不建议先建一张“销售待办主表”长期存储所有事项。

更稳的做法是：

- 从真实业务对象实时聚合待办
- 动作和节奏信息单独落表

---

## 三、V1 需要的最小表

V1 建议只新增或明确以下 3 类数据层：

1. `sales_follow_up_logs`
2. `sales_customer_reply_logs`
3. 可选：`sales_follow_up_plans`

不建议 V1 先新增：

- `sales_todos`
- `sales_task_queue`
- `unified_tasks`

原因：

- 这会把业务员待办又做成通用任务系统
- 容易偏离“从真实业务节点实时聚合”的原则

---

## 四、核心表 1：sales_follow_up_logs

## 4.1 用途

记录业务员所有真实跟进行为。

例如：

- 已联系客户
- 已发 QT
- 已催合同
- 已催定金
- 已发验货报告
- 已催付款
- 已通知到港
- 已催清关资料
- 已确认收货
- 已邀请反馈
- 约下次跟进

## 六、可选表 3：sales_follow_up_plans

## 6.1 是否必须

V1 可选，不是必须。

如果你们希望“约下次跟进”不是只挂在日志上，而是有独立计划结构，可以加这张表。

如果希望先轻实现，也可以只把 `next_follow_up_at` 放在日志表里。

---

## 6.2 用途

单独管理未来跟进计划，避免多条日志里的下次时间互相冲突。

## 七、待办生成规则

V1 不落单独待办主表，待办通过聚合实时生成。

## 7.1 聚合输入对象

| 对象 | 说明 |
|---|---|
| `Inquiry` | `ING` 阶段 |
| `QR` | 内部成本回传阶段 |
| `SalesQuotation` | `QT` 阶段 |
| `SalesContract` | `SC` 阶段 |
| `purchase_order_execution` | 完货后客户协同 |
| `voyage_tracking` | 在途 |
| `arrival_notices` | 到港 |
| `import_clearance_coordination` | 清关资料配合 |
| `delivery_confirmations` | 收货确认 |
| `post_order_feedback` | 客户反馈 |
| `sales_follow_up_logs` | 最近联系、下次跟进 |
| `sales_customer_reply_logs` | 最近客户回复 |

---

## 7.2 待办生成规则建议

### A. ING

触发：

- `ING` 已提交
- 指派给当前业务员
- 状态仍为待处理

生成待办：

- 新询价待响应
- 询价待澄清

### B. QR

触发：

- 已创建 `QR`
- 成本未回传

生成待办：

- 待内部成本反馈

### C. QT

触发：

- `QT` 已批准但未发送
- `QT` 已发送客户未反馈
- `QT` 协商中

生成待办：

- 待发报价
- 待客户确认报价
- 待跟进客户改价意见

### D. SC

触发：

- `SC` 已批准未发送
- `SC` 已发未签
- `SC` 已确认未付定金

生成待办：

- 待发合同
- 待签合同
- 待客户定金

### E. 完货后客户协同

触发：

- 完货后未确定验货方式
- 客户第三方验货未安排
- 第三方监装未安排
- 客户付款未完成

生成待办：

- 待确认验货方式
- 待安排第三方验货
- 待安排第三方监装
- 待客户付款

### F. 交付后协同

触发：

- 到港通知已发未确认
- 清关资料未配合
- 收货未确认
- 客户反馈未提交

生成待办：

- 待确认到港
- 待清关资料
- 待收货回执
- 待客户反馈

## 八、优先级规则

待办优先级建议分四级：

- `critical`
- `high`
- `medium`
- `normal`

### 1. critical

- 已超期未联系
- 尾款影响放单
- 到港后客户无回应

### 2. high

- 新 `ING`
- `QT` 已发多日未反馈
- `SC` 已发多日未签
- 客户承诺付款但未兑现

### 3. medium

- 一般跟进事项
- 待确认客户回复

### 4. normal

- 计划性复联
- 非阻塞型反馈邀请

---

## 九、V1 是否需要单独待办缓存表

结论：

- **不建议 V1 新增单独待办缓存表**

原因：

1. 会让业务员待办变成通用任务系统
2. 容易和工作台、其它角色待办再次混掉
3. V1 的数据量完全可以实时聚合

仅在以下情况才考虑 V1.1 / V2 增加缓存表：

- 业务员数据量非常大
- 页面实时聚合性能明显不足
- 需要复杂 SLA 批量计算

---

## 十、V1 最小可落地范围

V1 只强制落以下数据链：

1. `sales_follow_up_logs`
2. `sales_customer_reply_logs`
3. 实时聚合 `ING/QT/SC`
4. 实时聚合付款/到港/收货/反馈节点

这就足够支撑：

- 顶部动作条
- 主执行表
- 最近客户回复
- 最近跟进记录
- 详情抽屉动作

---

## 十一、后续扩展

V1.1 可以再加：

- `sales_follow_up_plans`
- 客诉协同子表
- 清关资料回执子表
- 第三方验货/监装专门协同表

V2 再考虑：

- 统一任务缓存
- SLA 计算
- 批量催办规则

---

## 十二、最终判断标准

只要这个模型能支撑以下 4 件事，就说明方向正确：

1. 能从真实业务链生成业务员待办
2. 能记录业务员真实跟进行为
3. 能记录客户真实回复
4. 能把关键动作回写到业务节点

如果模型开始变成“所有角色共用的万能任务系统”，就说明已经走偏。
