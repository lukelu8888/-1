# 正式对象编码清单

本文档用于固定当前 ERP 的正式业务对象编码口径。

目标：
- 避免 `ING / INQ / RFQ` 等历史混用再次出现
- 避免 `PI` 重新进入正式业务链
- 为后续流程、页面、待办、报表、单证、日志提供统一命名依据

---

## 一、正式主链

当前系统正式业务主链统一为：

`ING -> QR -> XJ -> BJ -> QT -> SC -> PR -> CG -> Shipment -> BL -> Finance Packet / Archive`

说明：
- `SC -> PR -> CG` 是唯一标准路径
- 不允许正式链路直接写成 `SC -> CG`
- `PI` 不再进入正式链路

---

## 二、正式对象编码

### 1. `ING`

- 中文名称：客户询价单
- 英文名称：Inquiry
- 所属侧：客户侧 / 业务员侧
- 责任起点角色：客户、业务员
- 作用：
  - 承接客户原始询价
  - 作为销售前链起点
  - 为后续 `QR / QT / SC` 提供原始需求来源
- 正式口径：
  - 所有新链路统一使用 `ING`
- 禁止再用：
  - `INQ`
  - `RFQ`

### 2. `QR`

- 中文名称：报价请求单 / 成本询报单
- 英文名称：Quotation Request
- 所属侧：业务员侧 / 采购侧
- 责任起点角色：业务员
- 作用：
  - 业务员将 `ING` 转成内部成本询报
  - 作为采购侧询价工作的起点
  - 为后续 `XJ / BJ / QT` 提供成本链主承载

### 3. `XJ`

- 中文名称：采购询价单
- 英文名称：Supplier Inquiry Request
- 所属侧：采购侧 / 供应商侧
- 责任起点角色：采购员
- 作用：
  - 采购基于 `QR` 向供应商发起询价
  - 供应商侧正式工作的起点
- 正式口径：
  - 一律使用 `XJ`
- 禁止再用：
  - `RFQ`

### 4. `BJ`

- 中文名称：供应商报价单
- 英文名称：Supplier Quotation
- 所属侧：供应商侧 / 采购侧
- 责任起点角色：供应商
- 作用：
  - 供应商对 `XJ` 的正式报价回应
  - 为采购比价、定厂提供依据
- 正式口径：
  - 供应商报价统一称 `BJ`
- 禁止模糊写法：
  - “供应商报价反馈”
  - “RFQ报价”

### 5. `QT`

- 中文名称：报价单
- 英文名称：Quotation
- 所属侧：业务员侧 / 客户侧
- 责任起点角色：业务员
- 作用：
  - 业务员基于 `QR / BJ` 形成正式对客报价
  - 推动客户确认并进入 `SC`
- 正式口径：
  - 只认 `QT`

### 6. `SC`

- 中文名称：销售合同
- 英文名称：Sales Contract
- 所属侧：业务员侧 / 客户侧 / 内部管理侧
- 责任起点角色：业务员
- 作用：
  - 销售承诺正式落地
  - 作为内部采购和后续履约的销售依据

### 7. `PR`

- 中文名称：采购申请单
- 英文名称：Purchase Requisition
- 所属侧：采购侧
- 责任起点角色：业务员下推，采购承接
- 作用：
  - 将 `SC` 转成内部采购需求
  - 做采购拆分、分供应商、分批次、数量/交期控制
  - 作为 `CG` 的直接上游
- 正式规则：
  - `SC` 后必须先有 `PR`

### 8. `CG`

- 中文名称：采购合同
- 英文名称：Procurement Contract
- 所属侧：采购侧 / 供应商侧
- 责任起点角色：采购员
- 作用：
  - 对供应商的正式采购执行单据
  - 后续样品、生产、自检、验货、完货、集货、装柜的执行起点

### 9. `Shipment`

- 中文名称：出运 / 发运批次
- 英文名称：Shipment
- 所属侧：单证侧 / 仓库侧 / 业务员侧 / 客户侧
- 作用：
  - 承接订舱、集货、装柜、报关、开船、在途、到港、清关、收货
- 说明：
  - 一个订单可拆分多个 shipment batch

### 10. `BL`

- 中文名称：提单
- 英文名称：Bill of Lading
- 所属侧：单证侧 / 业务员侧 / 财务侧 / 客户侧
- 作用：
  - 开船后核心单据
  - 与银行交单、放单控制、客户提货直接相关

### 11. `Finance Packet`

- 中文名称：财务合规文件包
- 英文名称：Finance Compliance Packet
- 所属侧：财务侧
- 作用：
  - 承接 `D05 / D09 / D10 / D11 / D12 / D13 / D01`
  - 形成退税与归档主包

---

## 三、历史口径与兼容规则

### 1. `INQ`

- 状态：历史旧口径
- 处理原则：
  - 不再用于正式链路
  - 历史兼容读取时可识别，但展示层应统一显示为 `ING`

### 2. `RFQ`

- 状态：历史旧口径
- 处理原则：
  - 不再用于正式链路
  - 采购询价统一改为 `XJ`

### 3. `PI`

- 状态：退出正式业务链
- 当前定位：
  - 仅作为模板中心 / 工作台里的纪念型展示模板存在
  - 不参与正式业务流程
  - 不参与正式对象链
- 正式替代：
  - 对客报价统一使用 `QT`

---

## 四、禁止写法

以下写法后续不应再出现在正式业务流程、待办、状态流转、角色蓝图中：

- `INQ -> QT -> SC`
- `SC -> CG`
- `RFQ`
- `供应商报价反馈`（不写 `BJ`）
- `PI` 作为正式流程节点
- `SO -> PO` 代替 `SC -> PR -> CG`

---

## 五、推荐写法

### 销售前链

`ING -> QR -> XJ -> BJ -> QT -> SC`

### 销售转采购

`SC -> PR -> CG`

### CG 后履约

`CG -> 样品 / 封样 -> 生产 / 自检 -> 验货 -> 完货 -> 出运`

### 出运与交付

`Shipment -> BL -> 在途 -> 到港 -> 清关 -> 收货 -> 反馈`

### 财务归档

`BL / 收汇 / 结汇 -> Finance Packet -> 财务归档 -> 业务归档`

---

## 六、后续开发约束

后续任何功能开发前，必须先确认：

1. 它属于哪一个正式对象
2. 它处于主链哪一段
3. 它展示给哪个角色
4. 页面和日志中使用哪个正式编码

如与本文档冲突，以本文档为准。
