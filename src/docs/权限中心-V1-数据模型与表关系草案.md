# 权限中心 V1 数据模型与表关系草案

## 一、目标

本草案用于为新建的 **权限中心 V1** 提供可实施的数据模型基础。

目标：
- 不直接破坏旧图1、图2
- 为新权限中心提供独立可落库的数据结构
- 与企业主数据中心形成清晰边界
- 布局和结构参考大厂常见的 IAM / ERP 权限中心做法

---

## 二、设计原则

### 1. 角色、菜单、动作、数据范围、例外授权分层

不要把这些内容混进一张大表：
- 角色
- 模块
- 动作
- 数据范围
- 具体人员例外

应拆成独立层。

### 2. 企业主数据中心负责“人和账号”

权限中心不重复存：
- 员工主档
- 账号主档
- 公司资料

权限中心只引用：
- 人员 ID
- 账号 ID
- 角色 ID

### 3. 角色是主授权单元

主权限链：

`角色 -> 模块 -> 动作 -> 数据范围`

人员只是例外层，不是主层。

### 4. 人员例外授权单独成层

真实人员例外授权不能污染主权限矩阵。

### 5. 所有变更必须可审计

权限变更必须保留日志。

---

## 三、参考大厂的布局逻辑

大厂常见做法不是“一张矩阵管所有事”，而是：

1. `Role Directory`
2. `Resource / Menu Matrix`
3. `Action Policy`
4. `Data Scope Policy`
5. `User Exceptions`
6. `Audit Logs`

所以本方案也按这个逻辑拆。

---

## 四、建议表结构总览

权限中心 V1 建议新增以下表：

1. `permission_roles`
2. `permission_modules`
3. `permission_role_module_access`
4. `permission_role_module_actions`
5. `permission_role_module_scopes`
6. `permission_user_exceptions`
7. `permission_change_logs`

其中：
- 角色表：定义正式角色
- 模块表：定义权限中心识别的模块
- 模块访问表：解决菜单可见性
- 动作表：解决按钮/操作权限
- 范围表：解决数据范围
- 例外表：解决按人临时授权
- 日志表：解决变更审计

---

## 五、各表说明

## 1. `permission_roles`

### 作用
定义正式角色字典。

### 建议字段
- `id` UUID PK
- `role_code` TEXT NOT NULL UNIQUE
- `role_name_zh` TEXT NOT NULL
- `role_name_en` TEXT NULL
- `role_type` TEXT NOT NULL
- `is_internal` BOOLEAN NOT NULL DEFAULT true
- `requires_region` BOOLEAN NOT NULL DEFAULT false
- `default_scope` TEXT NOT NULL
- `allow_login` BOOLEAN NOT NULL DEFAULT true
- `is_system_role` BOOLEAN NOT NULL DEFAULT true
- `description` TEXT NULL
- `status` TEXT NOT NULL DEFAULT `active`
- `sort_order` INTEGER NOT NULL DEFAULT 0
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `role_type`
  - `executive`
  - `management`
  - `business`
  - `support`
  - `external`
- `default_scope`
  - `all`
  - `region`
  - `department`
  - `subordinate`
  - `own`
  - `assigned`
- `status`
  - `active`
  - `inactive`

### 说明
- V1 先录入正式角色清单
- 不建议第一版做完全自定义角色工厂

---

## 2. `permission_modules`

### 作用
定义权限中心识别的菜单/模块。

### 建议字段
- `id` UUID PK
- `module_code` TEXT NOT NULL UNIQUE
- `module_name_zh` TEXT NOT NULL
- `module_name_en` TEXT NULL
- `module_category` TEXT NOT NULL
- `parent_module_code` TEXT NULL
- `module_level` INTEGER NOT NULL DEFAULT 1
- `route_key` TEXT NULL
- `description` TEXT NULL
- `status` TEXT NOT NULL DEFAULT `active`
- `sort_order` INTEGER NOT NULL DEFAULT 0
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `module_category`
  - `core`
  - `business`
  - `finance`
  - `supply_chain`
  - `system`
  - `support`
- `status`
  - `active`
  - `inactive`

### 说明
- 对应 AdminDashboard 的主导航和核心子模块
- 图2里的模块矩阵数据来源应来自这张表

---

## 3. `permission_role_module_access`

### 作用
定义角色对模块的“菜单可见性”。

### 建议字段
- `id` UUID PK
- `role_id` UUID NOT NULL FK -> `permission_roles.id`
- `module_id` UUID NOT NULL FK -> `permission_modules.id`
- `is_visible` BOOLEAN NOT NULL DEFAULT false
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### 唯一建议
- `UNIQUE (role_id, module_id)`

### 说明
- 这张表承接旧图2的主要能力
- 只负责“看不看得见”

---

## 4. `permission_role_module_actions`

### 作用
定义角色在某模块内可以执行哪些动作。

### 建议字段
- `id` UUID PK
- `role_id` UUID NOT NULL FK -> `permission_roles.id`
- `module_id` UUID NOT NULL FK -> `permission_modules.id`
- `action_code` TEXT NOT NULL
- `is_allowed` BOOLEAN NOT NULL DEFAULT false
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `action_code`
  - `view`
  - `create`
  - `edit`
  - `delete`
  - `approve`
  - `export`
  - `print`
  - `manage`

### 唯一建议
- `UNIQUE (role_id, module_id, action_code)`

### 说明
- 这张表吸收旧图1里动作图标表达的能力
- 动作层不应混回菜单矩阵主表

---

## 5. `permission_role_module_scopes`

### 作用
定义角色在某模块内的数据范围。

### 建议字段
- `id` UUID PK
- `role_id` UUID NOT NULL FK -> `permission_roles.id`
- `module_id` UUID NOT NULL FK -> `permission_modules.id`
- `scope_code` TEXT NOT NULL
- `scope_rules` JSONB NOT NULL DEFAULT `'{}'::jsonb`
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `scope_code`
  - `all`
  - `region`
  - `department`
  - `subordinate`
  - `own`
  - `assigned`

### 唯一建议
- `UNIQUE (role_id, module_id)`

### JSONB 用途
`scope_rules` 用来存不稳定的范围扩展：
- 指定区域列表
- 指定部门列表
- 指定审批线
- 特殊例外说明

### 说明
- 这张表非常关键
- 外贸 ERP 的“看哪些数据”比“能不能看菜单”更重要

---

## 6. `permission_user_exceptions`

### 作用
定义真实人员的临时/例外授权。

### 建议字段
- `id` UUID PK
- `user_profile_id` UUID NOT NULL
- `role_id` UUID NOT NULL FK -> `permission_roles.id`
- `module_id` UUID NOT NULL FK -> `permission_modules.id`
- `exception_type` TEXT NOT NULL
- `action_overrides` JSONB NOT NULL DEFAULT `'{}'::jsonb`
- `scope_override_code` TEXT NULL
- `scope_override_rules` JSONB NOT NULL DEFAULT `'{}'::jsonb`
- `effective_from` TIMESTAMPTZ NULL
- `effective_to` TIMESTAMPTZ NULL
- `reason` TEXT NULL
- `approved_by` TEXT NULL
- `status` TEXT NOT NULL DEFAULT `active`
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `exception_type`
  - `grant`
  - `restrict`
- `scope_override_code`
  - `all`
  - `region`
  - `department`
  - `subordinate`
  - `own`
  - `assigned`
- `status`
  - `active`
  - `inactive`
  - `expired`

### 说明
- 真实人员只在这张表里出现
- 这张表是图1中“按人差异”的落地层

---

## 7. `permission_change_logs`

### 作用
记录权限配置的变更日志。

### 建议字段
- `id` UUID PK
- `change_type` TEXT NOT NULL
- `target_type` TEXT NOT NULL
- `target_key` TEXT NOT NULL
- `before_snapshot` JSONB NOT NULL DEFAULT `'{}'::jsonb`
- `after_snapshot` JSONB NOT NULL DEFAULT `'{}'::jsonb`
- `changed_by` TEXT NULL
- `change_reason` TEXT NULL
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()

### CHECK 建议
- `change_type`
  - `create`
  - `update`
  - `delete`
- `target_type`
  - `role`
  - `module_access`
  - `module_action`
  - `module_scope`
  - `user_exception`

### 说明
- 权限中心必须可审计
- 这张表不是可选项

---

## 六、与现有表的关系

权限中心不应直接复制已有人员主数据，而是关联现有体系：

- 企业主数据中心中的内部账号
- `user_profiles`
- 组织架构/区域字段

建议关联方式：
- `permission_user_exceptions.user_profile_id`
  - 关联真实员工账号

---

## 七、推荐页面与数据表映射

### 1. `角色总览`
主要读：
- `permission_roles`

### 2. `菜单权限矩阵`
主要读写：
- `permission_modules`
- `permission_role_module_access`

### 3. `动作与数据范围`
主要读写：
- `permission_modules`
- `permission_role_module_actions`
- `permission_role_module_scopes`

### 4. `人员例外授权`
主要读写：
- `permission_user_exceptions`
- `user_profiles`

### 5. `变更日志`
主要读：
- `permission_change_logs`

---

## 八、V1 实施顺序建议

### 第一批
- `permission_roles`
- `permission_modules`
- `permission_role_module_access`

用于先落菜单矩阵主干。

### 第二批
- `permission_role_module_actions`
- `permission_role_module_scopes`

用于吸收旧图1中的动作与数据范围能力。

### 第三批
- `permission_user_exceptions`
- `permission_change_logs`

用于吸收按人差异和变更审计。

---

## 九、V1 暂不做

V1 不建议先做：
- 字段级权限
- 行级动态表达式权限引擎
- 外部侧统一权限中心
- 多租户超复杂授权继承

先把我方侧角色、模块、动作、范围和例外授权做稳定。

---

## 十、结论

权限中心 V1 的正确数据模型应该是：

`角色`
→ `模块访问`
→ `动作权限`
→ `数据范围`
→ `人员例外`
→ `变更日志`

这套模型既能吸收图1和图2的能力，也不会把所有内容再次混成一张超大矩阵。
