# 超时升级调度方案

## 结论

当前仓库将 `deadlineEscalationService` 的后台调度方案固定为：

- `外部 cron`

优先顺序：

1. `GitHub Actions schedule`
2. `Vercel Cron`
3. 其他可定时触发 HTTP/脚本任务的平台

## 不选用的方案

### `pg_cron`

暂不采用，原因：

- 当前 Supabase migration 中没有 `pg_cron` 扩展安装与治理约束
- SQL 任务不利于复用现有 TypeScript 业务规则
- 后续维护成本更高

### `Supabase Edge Function + cron`

暂不作为第一选择，原因：

- 仍需要额外配置调度器
- 当前业务规则已沉淀在前端/共享 TypeScript 服务中，先走外部 cron 更容易复用

## 运行策略

- 调度频率：`每小时一次`
- 运行入口：调用统一的“超时升级执行入口”
- 规则来源：`src/lib/services/deadlineEscalationService.ts`
- 执行入口：`src/lib/services/deadlineEscalationExecutionService.ts`

## 首批要执行的升级规则

- `ING 24h 未首响`
- `QT 7天未反馈`
- `SC 3天未签`
- `SC 定金超期`
- `O/A 账期 T-7`
- `DA 到期 T-3`
- `Usance L/C 到期 T-5`

## 落地要求

- 调度层只负责“定时触发”
- 业务判断统一走 `deadlineEscalationService`
- 通知写入统一落到通知/审批/待办体系，不在调度脚本里重复实现业务规则

## 后续接入建议

  - `.github/workflows/deadline-escalation.yml`
  - `scripts/run-deadline-escalation.mjs`

  - 本地真实执行入口：`npm run deadline:escalation`

## 环境与 secrets 校验

生产链路默认要求以下配置：

- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `DEADLINE_ESCALATION_RETRY_COUNT`，默认 `2`
- `DEADLINE_ESCALATION_DEDUPE_HOURS`，默认 `12`

说明：

- `scripts/guard-deadline-escalation-env.mjs` 会在非 fixture 模式下强校验 `SUPABASE_URL` 与 `SUPABASE_SERVICE_ROLE_KEY`
- fixture 模式允许在没有 Supabase secrets 时执行，用于本地演练与 CI 自检
- `DEADLINE_ESCALATION_DEDUPE_HOURS < 1` 或 `DEADLINE_ESCALATION_RETRY_COUNT < 0` 会直接失败

建议 GitHub 仓库 secrets / variables：

| 类型 | 名称 | 用途 |
|---|---|---|
| Secret | `SUPABASE_URL` | 生产环境 Supabase URL |
| Secret | `SUPABASE_SERVICE_ROLE_KEY` | 写通知去重与入队所需 service role |
| Variable | `DEADLINE_ESCALATION_RETRY_COUNT` | 通知写入失败后的重试次数，建议 `2` |
| Variable | `DEADLINE_ESCALATION_DEDUPE_HOURS` | 重复通知去重窗口，建议 `12` |

## 去重、重试与日志约束

当前链路已补齐以下生产保障：

- 重复通知检查：
  - 按 `referenceId + notificationType + recipientEmail + dedupe window` 做重复通知防抖
  - 在去重窗口内重复命中的事项将返回 `deduped`
- 失败重试：
  - 通知写入失败后按 `DEADLINE_ESCALATION_RETRY_COUNT` 重试
  - 当前默认重试 `2` 次
- 日志输出：
  - 调度脚本固定输出 `dryRun / useFixture / dedupeHours / retryCount / scanned / enqueued / deduped / dryRunCount / failures`
  - 每条事项会进入 `items`
  - 每条收件动作会进入 `dispatchResults`
  - 按规则汇总进入 `ruleSummary`

## 当前执行结果契约

调度脚本与执行服务已统一输出以下结果层级：

- `dryRun`：本轮是否为无副作用演练
- `items`：本轮扫描命中的升级事项
- `ruleSummary`：按规则类型聚合的扫描结果
- `dispatchResults`：每个事项每个收件人的写入结果（`enqueued / deduped / failed`）
- `failures`：失败明细，按 `duplicate_check / enqueue` 区分阶段

这意味着：

- 调度器只需要关注“是否成功执行”
- 业务侧可以继续复用统一的结果结构做日志、告警或后续监控

- 如果后续迁移到 Vercel Cron：
  - 保持同样的 hourly 节奏
  - 不改业务规则层
  - 只替换触发器
